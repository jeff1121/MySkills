# MySkills 專案憲章

## 核心原則

### I. 程式碼品質

**必須遵守**:
- 程式碼 MUST 通過 linter 與 formatter 檢查後才能合併
- 函式/方法 MUST 保持單一職責，長度不超過 50 行
- 命名 MUST 清晰表達意圖，避免縮寫（除非是業界通用術語）
- 註解 SHOULD 解釋「為什麼」而非「做什麼」

**理由**: 清晰、一致的程式碼降低維護成本，提升團隊協作效率。

### II. 測試標準

**必須遵守**:
- 核心業務邏輯 MUST 有單元測試覆蓋
- API 端點 MUST 有契約測試 (contract test) 確保輸入輸出規格
- 測試 MUST 可獨立執行，不依賴外部服務或特定執行順序
- Bug 修復 SHOULD 附帶重現該 bug 的測試案例

**理由**: 測試是品質的守門員，確保變更不會破壞既有功能。

### III. 使用者體驗一致性

**必須遵守**:
- UI 元件 MUST 遵循既定的設計規範與元件庫
- 錯誤訊息 MUST 對使用者友善，提供明確的下一步指引
- 互動回饋 MUST 在 100ms 內給予視覺反應
- 關鍵操作 MUST 有確認機制，避免誤操作

**理由**: 一致的體驗降低使用者學習成本，提升滿意度與信任感。

### IV. 效能要求

**必須遵守**:
- API 回應時間 MUST 在 p95 < 500ms（一般操作）
- 頁面首次內容繪製 (FCP) SHOULD < 1.5 秒
- 記憶體使用 MUST 無明顯洩漏（長時間執行測試驗證）
- 大量資料操作 MUST 支援分頁或串流處理

**理由**: 效能直接影響使用者體驗與系統可擴展性。

## MVP 開發原則

**核心精神**: 不過度設計 (YAGNI - You Aren't Gonna Need It)

- 功能 MUST 從最小可行版本開始，驗證價值後再擴展
- 抽象層 MUST 在有明確重複需求時才引入
- 架構決策 SHOULD 可逆轉，避免過早鎖定
- 每個 User Story MUST 可獨立交付與測試

**複雜度控管**:
| 如果你想要... | 先問自己... |
|--------------|------------|
| 新增抽象層 | 有 3+ 處重複嗎？ |
| 引入新依賴 | 現有工具無法滿足嗎？ |
| 增加配置選項 | 使用者真的需要嗎？ |

## 開發流程

**程式碼審查**:
- 所有變更 MUST 經過至少一人審查
- 審查 MUST 確認符合憲章原則
- 複雜度增加 MUST 在 PR 描述中說明理由

**品質關卡**:
1. Linter/Formatter 通過
2. 測試全數通過
3. 無新增 TODO 未標註負責人與期限

## 治理

本憲章為專案開發的最高指導原則，所有開發實踐 MUST 與憲章一致。

**修訂流程**:
1. 提出修訂提案並說明理由
2. 團隊討論並達成共識
3. 更新憲章版本號與日期
4. 同步更新受影響的範本與文件

**版本規則**:
- MAJOR: 原則移除或根本性變更
- MINOR: 新增原則或重大擴充
- PATCH: 措辭調整、錯字修正
